# MTGO Log Structure — Research Brief for Tamiyo Phase 2.5

Synthesized from five Khriss research notes (filed 2026-04-18) + Pattern cross-check.
No prior KB coverage existed — this is all net-new. Sources: cderickson/MTGO-Tracker,
BigPeet/mtgo_utils, videre-project/MTGOSDK, MTGO forums, MTGO EULA.

---

## 1. File locations — what the watcher should target

The current watcher default path (`%LOCALAPPDATA%\Apps\2.0\`) is correct but incomplete.
MTGO uses ClickOnce — the actual log files sit two random GUID-named subdirectories deep:

```
C:\Users\<User>\AppData\Local\Apps\2.0\<random>\<random>\mtgo..tion_<hash>\
```

**There is no stable absolute path.** The path changes on clean reinstall or forced client
update. This means the current hardcoded fallback in `config.py` (`Path(os.environ.get("LOCALAPPDATA", "")) / "Apps" / "2.0"`) is correct as the *search root*, but the watcher must glob-discover the actual subdirectory.

**Required change for Phase 2.5:**

At agent startup (and after any file-not-found error), glob `%LOCALAPPDATA%\Apps\2.0\`
recursively for files matching `Match_GameLog_*.dat`. Cache the discovered directory.
Re-run discovery if the cache goes stale. This is exactly what cderickson/MTGO-Tracker does.

**File patterns to watch:**

| File pattern | Location | Purpose |
|---|---|---|
| `Match_GameLog_*.dat` | `%LOCALAPPDATA%\Apps\2.0\<random>\<random>\` | Primary match log (binary) |
| `DraftLog*.txt` | `C:\Users\<User>\Documents\` | Draft session log (plaintext) |
| `mtgo_game_history` | Same AppData dir as .dat | Match history index — not a log |

Watchdog's `recursive=True` on `%LOCALAPPDATA%\Apps\2.0\` is the right approach — it's
already set in the current `watcher.py`. The only gap is that the watch root needs to be
the `Apps\2.0` parent, not a hardcoded full path.

DraftLog tracking is out of Phase 2.5 scope — it's a separate directory and format.

## 2. Text log parsing strategy

**MTGO only writes `.dat` files.** There is no companion plaintext `.log` automatically
generated by the MTGO client. The `.log` form only exists if an external tool (BigPeet/mtgo_utils)
has already converted the `.dat`. The current `parse_text_log` fallback in Tamiyo will never
fire on a raw MTGO installation unless BigPeet's tool is already running separately.

This changes the architecture: **Tamiyo must either parse `.dat` or convert it first.**
See Section 3 for the recommended approach.

### Log format (once plaintext is obtained)

These patterns come from direct inspection of real MTGO log data (BigPeet test data) and
cderickson/modo.py. They have been stable since July 2019 — the last known breaking change.

**Game boundary detection:**

| Event | Line pattern |
|---|---|
| New game (dice roll) | `<Player> rolled a <N>.` — appears at top of each game block |
| Game start | `<Player> joined the game.` — triggers new game object |
| Play order | `<Player> chooses to play first.` / `chooses to not play first` |
| Hand size | `<Player> begins the game with seven cards in hand.` |
| Mulligan | `<Player> mulligans to six cards.` then `puts a card on the bottom...` |
| Turn marker | `Turn <N>: <PlayerName>` |
| Skip draw (on play) | `<Player> skips their draw step.` |
| Game end (concede) | `<Player> has conceded from the game.` |
| Game end (win) | `<Player> wins the game.` |
| Game end (disconnect) | `<Player> has lost the game due to disconnection.` |

**Card reference markup** (embedded in all action lines):

```
@[Card Name@:CardID,InstanceID:@]
```

- `CardID`: stable integer (MTGO catalog ID — same across games)
- `InstanceID`: per-game object ID (unique to that battlefield instance)
- Italic ability text: `@i(...)@i`

**Example lines:**
```
Opponent89 plays @[Mountain@:157352,543:@].
Player01 casts @[Soul-Scar Mage@:127796,544:@].
Player01 activates an ability of @[Verdant Catacombs@:69408,546:@] ( Search your library...).
```

**Player name extraction:** Split on `" joined the game"` to get name. Collect all unique
names from the file, then sort longest-first before any string matching — this prevents
substring bugs if one username is a prefix of another (the bug cderickson documented).

**What the log does NOT contain:**

- Timestamps (event ordering only)
- Format/event type (must come from filename, user config, or separate source)
- Decklists
- Life total changes (inferred from damage events)
- Explicit phase/step markers (no "combat phase" line — only attack declarations)
- Match metadata (event ID, round, opponent rating)

**Mulligan word-to-int mapping** (from cderickson/modo.py):

```python
{"seven": 7, "six": 6, "five": 5, "four": 4, "three": 3, "two": 2, "one": 1}
```

**Recommended event taxonomy for ParsedPlay.action_type:**

| Action | Trigger phrase |
|---|---|
| `land` | `plays @[` |
| `cast` | `casts @[` |
| `ability` | `activates an ability of` |
| `trigger` | `puts triggered ability from` |
| `draw` | `draws a card with` |
| `discard` | `discards @[` |
| `attack` | `is being attacked by` |
| `sacrifice` | `sacrifices @[` |

**Win condition detection — CRITICAL BUG IN CURRENT PARSER:**

`parse_text_log` currently matches `"wins the match"` and `"loses the match"`. These strings
**do not appear in real MTGO log files** (confirmed from direct BigPeet test data inspection).
The current parser will never fire on a real log. The actual signals are:

- `<Player> has conceded from the game.` — game-level loss
- `<Player> wins the game.` — game-level win
- `<Player> has lost the game due to disconnection.` — game-level loss

**There is no single "wins the match" sentinel.** Match result must be inferred by tallying
game wins across all games in the file (BO3: first to 2 game wins wins the match). The
`WINS_PATTERN` and `LOSES_PATTERN` regexes in the current `parser.py` must be replaced
entirely. Also handle special win conditions via card abilities (e.g., Thassa's Oracle) —
those appear as triggered ability text, not a standard win/concede line.

## 3. .dat parser — recommendation: defer via subprocess

**Recommendation: defer the binary .dat parser stub, but implement the BigPeet bridge.**

The current `parse_dat_file` stub should NOT be fully reverse-engineered. Here's why:

- No published format spec — pure reverse engineering
- The only working parser (BigPeet/mtgo_utils) is opaque C++ with no documented internals
- The format can break on any MTGO client update with no notice
- The effort is high, fragile, and unnecessary

**What to do instead (the bridge approach):**

Call `BigPeet/mtgo_utils file_parser` as a subprocess to convert `.dat` → plaintext, then
feed the plaintext through `parse_text_log`. The binary parsing problem disappears entirely.

```python
import subprocess

def convert_dat_to_text(dat_path: Path, file_parser_bin: Path) -> str | None:
    result = subprocess.run(
        [str(file_parser_bin), str(dat_path)],
        capture_output=True, text=True, timeout=10
    )
    if result.returncode != 0:
        logger.warning("file_parser failed on %s: %s", dat_path, result.stderr)
        return None
    return result.stdout
```

`file_parser` is MIT-licensed and ships Windows binaries — bundle it with Tamiyo's
PyInstaller package. The binary lives at `agent/vendor/file_parser.exe`.

**Fallback behavior (if file_parser not bundled or fails):**

The current behavior — return `None` from `parse_dat_file`, fall through to `parse_text_log` —
is the right design. But since MTGO never writes plaintext logs itself, the fallback produces
nothing. The correct fallback chain for Phase 2.5:

1. Try `file_parser` subprocess → parse resulting plaintext (rich result)
2. If subprocess fails → log warning, return `None` (match goes unrecorded)
3. Do NOT try `parse_text_log` on a `.dat` path — it's binary, not text

The existing `parse_file()` dispatch function needs updating to implement this chain.

**What the bridge gives Phase 2.5:**

Full game-level and play-level data — `ParsedGame` and `ParsedPlay` can be populated,
not just the top-level `ParsedMatch`. This is the blocker for real match ingest.

## 4. Inspiration repo — cderickson/MTGO-Tracker

**Repo**: [github.com/cderickson/MTGO-Tracker](https://github.com/cderickson/MTGO-Tracker)
**Status**: RETIRED — author directs users to Mox-Data.com. Not maintained.

**What to borrow:**

The parsing logic in `modo.py` is the gold standard. Key functions/patterns to study and adapt:

| What | From modo.py | Notes |
|---|---|---|
| File discovery | Glob `Apps\2.0\` for `"Match_GameLog_"` in filename, len > 30 | Direct port |
| Player name extraction | `line.split(" joined the game")[0]` | Port directly |
| Game boundary detection | `"joined the game"` → new game, `"rolled a"` → match start | Port directly |
| Card reference parsing | Split on `@[` and `@]` to extract name + IDs | Port directly |
| Mulligan counting | `"begins the game with X cards in hand"` → word-to-int | Port directly |
| Turn tracking | `"Turn X:"` prefix | Port directly |
| Play order | `"chooses to play first"` | Port directly |
| Win detection | `"has conceded"`, `"has lost the game"`, `"wins the game"` + card conditions | Port + extend |
| Name substring protection | Sort names longest-first before matching | Port directly |

**What NOT to borrow:**

- SQLite storage model — Tamiyo has PostgreSQL with a defined schema
- The overall application structure — Tamiyo's architecture is different
- The event ID extraction from filename (`file.split('-')[-2]`) — confirmed valid approach,
  use this for `mtgo_match_id` (see stable ID note below)

**Stable match ID:** The current Tamiyo text parser generates a synthetic ID
(`text-<stem>-<uuid8>`) that is NOT stable — the same file re-parsed produces a different ID,
breaking dedup. The fix: extract the match/event ID from the `.dat` filename using
`filename.split('-')[-2]` (the cderickson approach, confirmed against real filenames).
This is stable across re-parses of the same file and unique per match.

**The retired status is not a concern for parsing logic** — the format has been stable since
2019, and modo.py's patterns are grounded in real log inspection. The retirement risk is
that future MTGO format changes break the patterns without anyone maintaining them, which
is a general risk for all log-based approaches (see Section 5).

## 5. Risks to design around

In priority order:

**Medium — ClickOnce path instability** — path changes on reinstall. Mitigate: dynamic
discovery at startup and on file-not-found errors. Already designed for — watcher root
just needs to be `Apps\2.0` not a full hardcoded path.

**Medium — Log format breaking changes** — Daybreak does not publish a spec or changelogs
for log format. Last known break: July 2019. Mitigate: design the parser defensively —
log all unrecognized line patterns, emit a health-check warning if > N% of lines in a file
go unmatched, subscribe to mtgo.com/news for heads-up on client updates.

**Low-Medium — File lock / race conditions** — MTGO writes `.dat` during a live match.
Tamiyo's 2-second debounce (already in `watcher.py`) is the right mitigation. Also detect
incomplete game blocks (no concession/win line in the last game) and treat as in-progress —
do not ingest partial matches. Re-process after the next modification event.

**Low — ToS** — reading log files MTGO writes to disk is not explicitly prohibited.
Multiple community tools do this openly (MyMTGO, MTGO-Tracker) without enforcement action.
Stay passive: no process injection, no network interception, no memory reading.
The MTGOSDK (ClrMD process memory) approach is the risky one — do not use it.

**Low — Player name substring** — if one MTGO username is a prefix of another in the same
match, naive string matching corrupts extraction. Fix: sort names longest-first (cderickson's
documented fix). Already manageable but the current parser doesn't do this — add it.

**Low — Encoding** — `.dat` text portions appear UTF-8. Open with UTF-8, `errors="replace"`.
Card names are ASCII-safe. This is already what `parse_text_log` does.

## Implementation order for Phase 2.5

Given the above, here's the recommended sequence:

1. **Fix the broken win/loss regex** — `WINS_PATTERN` / `LOSES_PATTERN` in `parser.py`
   match strings that don't exist in real logs. Replace with game-level signals and
   match-result inference across games. This is the most urgent fix — current parser
   produces no results on real MTGO output.
2. **Fix `mtgo_match_id`** — replace `text-<stem>-<uuid8>` with `filename.split('-')[-2]`.
3. **Fix path discovery** — change watcher to glob `Apps\2.0\` for `Match_GameLog_*.dat`
   at startup rather than watching a hardcoded path.
4. **Bundle BigPeet file_parser** — add `agent/vendor/file_parser.exe` to the repo,
   add subprocess bridge in `parser.py`.
5. **Replace `parse_dat_file` stub** with the subprocess bridge.
6. **Upgrade `parse_text_log`** using cderickson/modo.py patterns: full game-boundary
   detection, turn tracking, play-level extraction, card reference parsing.
7. **Update the upload handler** (`app/api/agent.py`) to persist the full `ParsedMatch`
   including games and plays, completing the match ingest stub.

Steps 1-5 are agent-side (pure Python, no server changes). Step 6 is server-side and the
actual Phase 2.5 deliverable — the agent work feeds it.
