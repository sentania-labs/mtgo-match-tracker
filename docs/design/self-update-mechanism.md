# Self-update mechanism

Pointer doc for the Windows agent's self-update flow. Code lives in
`agent/updater.py` (network + verify + exec handoff) and `agent/tray.py`
(background loop, staged state, menu).

## Update source

GitHub Releases, repo `sentania-labs/mtgo-match-tracker`. Agent polls
`GET https://api.github.com/repos/.../releases/latest` at startup and
every `updates.check_interval_hours` (default 1h). If
`updates.include_prereleases` is false, prereleases are skipped. An
optional `updates.github_token` is attached for private repos or to
avoid anonymous rate limits.

The release must carry two assets named exactly:

- `MTGOMatchTracker.exe` — the new binary
- `MTGOMatchTracker.exe.sha256` — plaintext SHA256 of the exe

Both are produced by the `release.yml` workflow on tag push.

## Integrity check

`download_and_verify()` downloads the exe to
`%TEMP%\MTGOMatchTracker_update.exe`, fetches the companion
`.sha256` asset, and compares a fresh SHA256 of the downloaded file
against the published value. On mismatch the downloaded file is
deleted and the function returns `None` — the old agent keeps running
untouched.

The `.sha256` asset is fetched over HTTPS from the same release; if
an attacker can tamper with the release assets, they can tamper with
both the exe and the checksum. The checksum protects against in-flight
corruption and incorrect/partial downloads, not against a compromised
GitHub repo. Signing is future work if stronger guarantees are needed.

## Rollback on failure

There is no explicit rollback because nothing is installed until the
user clicks Restart to Update:

- Download failure → staged path stays `None`, old agent keeps running,
  user sees `Update {tag} download failed — check log.`
- Checksum failure → staged exe is unlinked, same failure path.
- `apply_update()` failure on Windows (launch raised) → exception
  logged, staged path stays, user can retry; old agent still owns the
  tray.

Because the new exe lives only in `%TEMP%` until the handoff, aborted
updates leave no persistent state behind beyond the temp file (which
Windows clears on its normal schedule).

## Restart handshake

Windows cannot replace a running exe in place. The handshake in
`apply_update()` side-steps this by launching the new exe as a detached
subprocess and exiting the old process:

```
subprocess.Popen([new_exe], close_fds=True)
raise SystemExit(0)
```

The new exe — which lives at a different path (`%TEMP%\...`) — then
takes over the tray role. Installing the new exe to the original
location (overwriting the old one) is the new exe's responsibility on
first launch; that logic is future work. For MVP the new exe runs from
`%TEMP%` after restart.

On non-Windows (dev/CI), `apply_update()` is a no-op that logs; the
tray clears its staged state and the test continues.

## Security considerations

- **HTTPS only** — GitHub API and asset URLs are https; httpx raises
  on mismatch by default.
- **Checksum-gated** — no code runs until the exe's SHA256 matches the
  signed-by-GitHub asset listing.
- **Trusted path** — the downloaded exe path (`%TEMP%\...`) is
  generated by the agent, not supplied by the server or the API
  response, so a malicious `browser_download_url` cannot redirect the
  handoff target outside the agent's control.
- **User-gated restart** — download is automatic, but `apply_update()`
  only runs when the user explicitly clicks Restart to Update.
- **Rate-limit / token** — `updates.github_token`, if set, is sent as
  a bearer on the API + asset fetch. It is never forwarded to the
  server.

## Operational flow (user perspective)

1. Agent launches, tray icon appears.
2. Updater thread checks for new release (startup + every N hours).
3. If a newer version exists and has the expected assets, the agent
   downloads it in the background. No notification until verify
   passes.
4. On success: tray notification `Update {tag} downloaded — click
   'Restart to Update' to apply.` A new menu item `Restart to Update`
   becomes visible.
5. User picks the menu item when ready. Old agent launches the new
   exe and exits; the tray icon flickers and the new agent takes
   over.
6. If the user never clicks: old agent keeps running. Each poll skips
   re-downloading the same tag (dedup by `_staged_update_tag`).
7. On failure (download or checksum): tray notification `Update {tag}
   download failed — check log.` No menu item, no restart.

Manual trigger: the existing `Check for Updates` menu item calls the
same `_check_updates_once()` path on a fresh daemon thread.

## Threading

All network I/O runs on the updater daemon thread (`mtgo-updater`) —
`asyncio.run()` blocks that thread until download + verify complete.
Menu rebuilds go through `self._icon.update_menu()`, which pystray
marshals back to the tray event loop. The staged path/tag pair is
guarded by `self._staged_lock` because the tray thread reads it when
`Restart to Update` is clicked.
